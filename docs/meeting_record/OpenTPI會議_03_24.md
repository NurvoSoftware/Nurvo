### 系統架構與專案進度會議記錄

#### 一、 系統架構決策：從「中央控管」改為「廣播式多 Agent 獨立判斷」
你們在會議中推翻了最初「用一個 Agent 當 Router」的設計，最終決定採用類似真實病房情境的**多 Agent 廣播模式**。

* **角色配置：** 1 位護理師（玩家/使用者）、1 位病患（Agent）、多位家屬（Agent，如 A, B, C）。
* **資訊流傳遞（核心機制）：**
    1.  護理師（玩家）說話後，透過 STT（語音轉文字）轉成文字。
    2.  這段文字會**同時廣播（Broadcast）**給病患與所有家屬 Agent。
    3.  各個 Agent 根據自身被設定的 System Prompt（角色背景、個性，例如：暴躁爸爸、溫柔媽媽），**獨立判斷**是否要回話。
    4.  Agent 回傳的結果需帶有 Boolean 參數（例如 `say: true` 或 `false`）。
    5.  只有判斷為 `true` 的文字，才會進入 ElevenLabs 進行 TTS（文字轉語音），並播放給護理師聽。同時，所有產出的對話紀錄都會即時顯示在前端的文字聊天室中。

#### 二、 遊戲機制與通關評分
放棄了繁瑣的「呼叫醫生」和「無止盡迴圈」，改用更明確的限制來計算成績。

* **核心目標：** 護理師必須在對話中，引導病患說出 **6 個關鍵健康指標**。
* **難度控制：** 難度差異**不在於**家屬插話的頻率，而是調整「病患 Agent 的 System Prompt」。難度越高，病患越不願意透露資訊，需要護理師更高超的溝通技巧。
* **通關條件與評分機制：**
    * **時間限制：** 設定固定時間（如 5 到 10 分鐘）。時間一到，對話強制結束。
    * **結算畫面：** 跳出表單，讓護理師填寫收集到的 6 個指標。
    * **評分依據：** 根據「答對的指標數量」與「花費的時間」交由公式或另一套 LLM 進行綜合評分（詳細評分邏輯待定，但資料會一併送入後端）。

#### 三、 開發實作細節與分工
* **前端（護理師介面）：**
    * 放棄耗時的 3D 模型，先以 **2D 介面（平面圖）** 實作。
    * 畫面右側需有文字聊天室，完整顯示所有 Agent 的發言（即使語音被中斷或沒播完，文字也要保留，避免玩家錯失資訊）。
    * 具備一個錄音/送出按鈕。玩家講話時，所有 Agent 的語音需自動閉嘴（打斷機制）。
* **後端與 API：**
    * 主要邏輯集中在後端。玩家發言後，後端需併發發送 API 給 4 個 LLM（病患 + 3 個家屬）。
    * 等待所有 LLM 回傳 JSON（包含 `say` 的狀態與 `text`）。
    * 負責將需要發聲的文字串接 ElevenLabs API 並回傳給前端。
    * 考慮使用 LangChain 等套件來管理各個 Agent 的 Memory。

#### 四、 專案時程
* **Demo 目標：** 清明節連假結束後需整合完畢，產出可執行的成品（不先向教授報告，直接拿成品 Demo）。
* **下次 Sync：** 暫定於週日或週一確認進度。

### 💡 顧問視角：你邏輯上的潛在漏洞與風險

你們最後決定讓所有 Agent 自己決定要不要講話（`say: true/false`），這個想法在還原真實場景上很棒，但在**工程實作與 UX 上會遇到災難**，你們在會議中沒有給出明確的解法：

1.  **語音重疊與競爭狀態 (Race Condition)：**
    如果護理師講了一句話，暴躁爸爸（Agent A）和焦慮媽媽（Agent B）的 LLM 都判定 `say: true`，且幾乎同時回傳。前端會怎麼播？
    * *狀況 A：* 兩個語音檔同時播放，玩家聽見一團噪音。
    * *狀況 B：* 後端沒有做好排隊機制，導致報錯。
    **修正建議：** 你們的後端必須實作一個 **「語音播放佇列 (Audio Queue)」**。當多個 Agent 決定發言時，依據角色優先級或回傳速度排序播放，或者讓文字先顯示在畫面上，語音依序講出。

2.  **LLM 判斷 `say: true` 的不可控性：**
    你們依賴 LLM 自己判斷要不要插話。如果 Prompt 沒寫好，極大機率會發生：
    * 所有家屬每一輪都回傳 `true`，導致遊戲變成吵架模擬器，時間內根本問不到病患。
    * 所有家屬都回傳 `false`，家屬機制形同虛設。
    **修正建議：** 雖然取消了中央 Router，但後端依然需要一個**強制閥值（Threshold）**。例如：設定「一輪對話中，最多只能有 1 位家屬發言」。後端收到 3 個家屬都想講話的請求時，由程式邏輯（亂數或權重）只挑選 1 個過關送去 TTS，其他的強制攔截。
    
#### 可能的解法
* 視覺洗版，語音單軌： 讓文字聊天室瘋狂跳出家屬 A、B、C 的干擾訊息（視覺焦慮），但語音系統一次只播一個人的聲音。
* 真實的「插話」機制： 當病患語音播到一半，如果暴躁家屬決定發言，後端直接卡斷病患的語音，強制切換成家屬的語音。這才是真正會讓人煩躁的「被打斷」，而且聽得清楚內容。
* 主次音量區分（Ducking）： 如果真的非得同時播，技術上前端必須把家屬的語音設定為極小聲的「背景碎念模式」，只有當前主要的對話對象音量是 100%。
