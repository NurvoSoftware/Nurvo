## 🚀 專案：醫病溝通 AI 模擬系統 (暫定)
**會議目標**：確立清明節前 MVP (最小可行性產品) 的畫面流程、技術選型與團隊分工。

---

### 📱 一、 系統介面與 UX 流程 (Frontend Design)
系統預計分為三個主要畫面：

1. **首頁 (Start Screen)**
   * **視覺設計**：大型 Logo (黑底或白底待定)，畫面上方會有霧狀或其他覆蓋物進行跑動式的動態效果。
   * **互動**：畫面下方設置一個進入按鈕，點擊後跳轉至關卡選擇頁面。
2. **關卡選擇 (Level Selection)**
   * **視覺設計**：目前決定採用最簡約的「方形方框」來代表各個關卡。
   * *(備註：曾討論過使用不同場景的圓形圖示散佈在畫面上，但考量目前開發進度，先以簡單方框為主)*。
3. **模擬對話畫面 (Main Gameplay / Simulation)**
   * **場景設定**：第一人稱視角，模擬真實醫療現場（例如：醫生看診，周圍有病患與家屬）。
   * **人物互動機制**：
     * 放棄傳統 Google Meet 的網格排版，改採「身歷其境」的彈出設計。
     * 當 AI (病患或家屬) 講話時，人物視窗會從對應的位置（如正前方或側邊）彈出。
     * **人物動畫**：目前先求有，只要人物在講話時嘴巴會動即可；更細緻的臉部表情同步列為後續優化項目。
   * **語音與文字介面**：
     * 畫面中央設置麥克風按鈕/語音輸入區塊（會有音量波動視覺效果），讓使用者直接用語音與 AI 對話，提升臨場緊張感。
     * **逐字稿顯示**：是否要在畫面上即時顯示 AI 與使用者的對話文字（逐字稿）目前尚在糾結，擔心會破壞沈浸感與排版。

---

### 🛠️ 二、 技術架構與選型 (Tech Stack)
為了趕在「清明節」前產出可運作版本，初期以串接 API 為主。

* **前端框架**：決定使用 **Vue** (取代 React，以便團隊成員能共同貢獻前端程式碼)。
* **後端與 AI 服務**：
  * 核心功能：需要整合 **LLM (大型語言模型)**、**STT (語音轉文字)**、**TTS (文字轉語音)**。
  * 開發輔助：團隊將大量仰賴 AI 工具 (如 GitHub Copilot, Claude, ChatGPT) 協助撰寫程式碼。
* **資料庫 (Database)**：**初期先不做**。考量到開發時程，清明節版本暫不串接 DB。未來再加入登入系統、個人資料與聊天紀錄儲存（供後續教授分析使用）。
* **資安與部署**：
  * 考慮到醫院資料落地與安全性，未來可能需要尋找開源模型並自行部署防護。
  * 暫不考慮完全依賴特定的第三方管理平台 (如提及的 DGR)，以避免依賴性過高或潛在的授權/盈利限制。

### 📋 三、 專案管理與分工 (Project Management)
* **協作方式**：
  * 使用 GitHub 進行版本控制，成員各自開 Branch 開發，完成後再發 PR (Pull Request) 合併。
* **初步分工**：
  * **前端**：1 人負責 Vue 頁面開發與介面串接。
  * **後端 (A)**：1 人負責 Event Loop / 基礎 API 串接。
  * **後端 (B)**：1 人負責 LLM (AI 模型) 的對接與邏輯。
